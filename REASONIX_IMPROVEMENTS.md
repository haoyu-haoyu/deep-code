## Executive summary

The cache-moat self-verification workstream is the right thing to do first because it protects DeepCode's core competitive claim — "we keep the DeepSeek prefix cache hot turn-over-turn" — which today is asserted only by comments (stable-prefix.mjs:9-13, deepseekHarnessPrompts.ts:26-31) and a single happy-path live script (scripts/deepseek-cache-e2e.mjs) that is NOT in CI and only checks `prompt_cache_hit_tokens > 0` on two identical requests. We have rich telemetry that HASHES the stable prefix (stable-prefix.mjs, cache-telemetry.mjs) but ZERO tests that drive the real request builder through a multi-turn tool loop and assert the wire-level byte-prefix is stable, and no measurement of the reasoning_content re-send cost that Reasonix flagged (~+500 prompt tokens/turn in realcache_test.go Probe 2).

Grounding is solid: every file in the plan exists and the line references are accurate. The test homes (test/deepcode-native.test.mjs, test/p2-3-cache.test.mjs) are already in the CI node --test list (.github/workflows/ci.yml L42-70) and already import buildDeepSeekRequest, runDeepSeekAgent, streamDeepSeekQuery, stable-prefix and cache-telemetry — so the mock e2e, compaction-collapse, boundary-alignment, and chokepoint-audit tests need NO workflow edit. streamDeepSeekQuery already accepts context.fetch, so a fetch-override mock is a first-class, supported path. mapMessagesToDeepSeek has exactly one production caller (buildDeepSeekRequest L148), making the reasoningReplay knob a clean, low-blast-radius thread.

The work splits cleanly into three value tiers. Tier 1 (P2.11.a, do first): the mock-DeepSeek prefix-stability e2e + compaction-collapse test + the transient-state chokepoint guard — these LOCK IN the moat in CI with zero new dependencies, zero API cost, and high confidence, and they will catch any future regression that silently breaks the cache. Tier 2 (P2.11.b): the tool-result backward-alignment fix in partialCompactConversation — a real correctness bug (an orphan tool_result whose assistant tool_use was summarized away breaks both the prefix AND DeepSeek's tool-call/result pairing) that Reasonix explicitly guards against (compact.go compactBounds L170-172) and we currently do not. Tier 3 (P2.11.c, gated): the reasoningReplay knob + the two LIVE probes — valuable but should ship behind data, default-preserving current behavior, with the live cost probe run manually before we ever consider flipping the default.

Honest call: the reasoningReplay knob is the lowest-confidence bet. Our own constants/deepseekHarnessPrompts.ts:12-14 deliberately re-send reasoning "so DeepSeek can continue the same reasoning trajectory" — that is a coherence decision, not an oversight. The probe must first confirm (a) the active model even EMITS reasoning_content on tool turns (v4-flash may not) and (b) the billed delta is material, before we touch the default. Build the knob + probe; do NOT flip the default in this roadmap.

## Phases

### P2.11.a — Make the cache moat self-verifying in CI (mock e2e, no API cost)
- Mock-DeepSeek prefix-stability e2e driving real runDeepSeekAgent through a multi-turn tool loop
- Compaction-only-mutation collapse test (byte-prefix breaks exactly once at the boundary)
- Transient-state chokepoint audit guard test (attachments/reminders ride the tail, never the system prefix)
- Document the 'compaction is the only prefix-mutation point' invariant (JSDoc + harness prompt)

_Rationale:_ This is the highest-value, lowest-risk tier and directly protects core positioning, so it goes first. All four land in test files already in the CI node --test list with zero new dependencies and zero live-API cost. The fetch-override mock is fully supported (streamDeepSeekQuery accepts context.fetch), and the test files already import every symbol needed. Ordering inside the phase: build the mock harness first (it is reused by the collapse case), then the audit guard, then write the invariant doc last so it can cite the now-passing guard test.

### P2.11.b — Compaction boundary correctness — backward-align off tool results
- alignKeepBoundaryOffToolResults pure helper + unit test (port of Reasonix compactBounds tool-walk)
- Wire alignment into partialCompactConversation slice (both 'from' and 'up_to')
- Verify REPL caller pivotIndex stays valid after alignment

_Rationale:_ A genuine correctness bug, not just a cache optimization: when compaction's keep-boundary lands on a tool_result whose originating assistant tool_use was summarized away, the kept tail starts with an orphan tool_result — which both mutates the prefix mid-stream AND violates DeepSeek's tool_call/tool_result pairing. Reasonix guards this explicitly (compact.go L170-172). It depends on the invariant being documented (P2.11.a) so the fix is framed correctly, and the mock harness from P2.11.a can validate the post-fix prefix behavior. Medium effort because partialCompactConversation has two slice directions and a live REPL caller to re-validate.

### P2.11.c — reasoning_content cost: measure first, knob second, flip never (this roadmap)
- reasoningReplayOnToolCalls config knob threaded through resolveDeepSeekConfig -> mapMessagesToDeepSeek -> normalizer (default true = current behavior)
- reasoning round-trip cost MOCK test (replay on vs off; both prefix-stable, replay adds fixed per-turn miss)
- Live env-gated reasoning-cost probe script (manual, not in CI)
- Extend/keep live cache-e2e as manual DEEPCODE_REAL_E2E probe

_Rationale:_ Lowest-confidence tier, deliberately last. The current re-send is an intentional coherence choice (deepseekHarnessPrompts.ts:12-14), so we build the knob default-preserving and gate any default-flip on live data. The mock cost test (CI) proves the knob WORKS and isolates the cost source without API spend; the live probe (manual) quantifies the real billed delta and checks whether the active model even emits reasoning_content. Knob plumbing must precede both tests since they toggle it. No default change ships in this roadmap.

**Target design:** TARGET DESIGN:

**Target design:** A new `src/cli/serve/acp/` directory (kept as `.mjs` to match the sibling serve modules, OR `.ts` if it must import typed query internals — see step notes) containing four pieces mirroring Reasonix's split but adapted to our TS/Ink + query() architecture:

1. `protocol.mjs` (port of Reasonix protocol.go / the referenced main `protocol.ts`): plain shape definitions + helpers, no I/O. `ProtocolVersion = 1`; JSON-RPC error codes; `flattenPrompt(blocks)` (text blocks + resource.text -> trimmed string); `toolKind(toolName, tool?)` classifier; the InitializeResult capability object (loadSession:true, promptCapabilities {image:false,audio:false,embeddedContext:true}, mcpCapabilities {http:false,sse:false}).

2. `jsonrpc.mjs` (port of Reasonix server.go Conn): a transport-agnostic NDJSON JSON-RPC 2.0 peer over a `{input: Readable, output: Writable}` pair. Methods: `handle(method, fn)`, `handleNotify(method, fn)`, `notify(method, params)`, `request(method, params, {signal})` (correlates integer ids, resolves on matching response, rejects on close/abort), `serve()` (reads NDJSON lines, dispatches requests/notifications to async handlers, requests/notifications run concurrently so a long prompt doesn't block cancel/permission). Write-serialization via an async mutex/queue (Node writes are not interleave-safe under backpressure — reuse the `once(res,'drain')` pattern from sse.mjs). stdout is the JSON-RPC channel; ALL logs/diagnostics must go to stderr (mirror the serve modules' discipline).

3. `dispatch.mjs` (port of Reasonix dispatch.go updateSink): an `AcpSink` bound to a sessionId that consumes DeepCode's `query()` Message stream (the SDK-shaped messages already produced by runHeadlessStreaming / messageToSDK) and emits `session/update` notifications. Mapping: assistant text content block -> `agent_message_chunk`; thinking/reasoning block -> `agent_thought_chunk`; assistant `tool_use` block -> `tool_call` update (toolCallId=tool_use.id, title=tool.userFacingName, kind=toolKind, status:'pending', rawInput=input); user `tool_result` block -> `tool_call_update` (status completed/failed, content clipped to ~8000 chars matching Reasonix `maxResultChars`); a `replay(messages)` method (for session/load) emits the same updates with status:'completed' and a `user_message_chunk` for user turns.

4. `service.mjs` (port of Reasonix service.go): owns the JSON-RPC handlers + per-connection session registry (Map<sessionId, {abortController, cwd, run}>). `initialize` returns capabilities + agentInfo {name:'deep-code', version from package.json}. `session/new` mints a UUID (`randomUUID`), records cwd, returns {sessionId}. `session/prompt` flattens blocks, then runs ONE agentic turn by invoking the SAME assembly print.ts uses — but with `canUseTool` bridged to ACP `session/request_permission` (see below) — consuming the resulting Message stream through the `AcpSink`, and returns `{stopReason}` (end_turn / cancelled / error). `session/cancel` aborts the session's in-flight AbortController. `session/load` resolves the JSONL via `resolveSessionPath`, parses it, and `AcpSink.replay`s it.

Permission bridge (the keystone, and why no refactor is needed): ACP `session/request_permission` is structurally identical to DeepCode's existing `can_use_tool` control_request. Build an ACP-flavored `CanUseToolFn` that, on each gated tool call, sends `session/request_permission` (params: sessionId, toolCall {toolCallId, title, kind, rawInput, status:'pending'}, options [allow_once, allow_always, reject_once]) and maps the `outcome` reply to a `PermissionDecision`: selected allow_once/allow_always -> `{behavior:'allow', updatedInput:input}`; reject/cancelled -> `{behavior:'deny', message:'Rejected by user'}`. allow_always additionally records a session-scoped allow (reuse the same `forceDecision`/session-allow plumbing the TUI uses) so the same tool isn't re-prompted. This reuses the proven `sendRequest`/`createCanUseTool` round-trip pattern verbatim — only the wire method name and option shape differ.

Wiring: `index.mjs` replaces the exit-78 stub: when `acp` is set, construct the JSON-RPC peer over (process.stdin, process.stdout), instantiate `service`, register handlers, and `await serve()`. The transport-agnostic split (jsonrpc.mjs taking input/output, sink taking a notify fn) is the "one transport-agnostic Controller" idea adapted from Reasonix WITHOUT a rewrite: we don't unify TUI/SSE/ACP under one controller; we make the ACP layer itself transport-agnostic and reuse print.ts's headless assembly as the engine.

## controller-desktop-bet  ·  priority=low value=medium

**Effort:** Phase 0+1 (the recommended scope: define wire, write toWireEvent, make serve real, flag-gate, tests): S-M, ~2-3 PRs. Phase 1d (TUI reducer convergence): M, +1 PR, optional. **Current state:** DeepCode has THREE consumers of the agent loop, with TWO divergent event vocabularies — the opposite of Reasonix's single-wire design:

**Target design:** Two distinct things are conflated in "the controller bet" — separate them, because their ROI is wildly different:

A. THE EVENT WIRE (cheap, high-leverage, do now): a single typed, serializable event vocabulary that the agent loop emits and EVERY frontend consumes. Today the TUI's `handleMessageFromStream` is effectively DeepCode's event mapper, but it (1) fans into React closures rather than producing values, and (2) is bypassed entirely by serve. Target: introduce a pure function `toWireEvent(message): DeepCodeEvent | null` (mirroring Reasonix's toWire) that maps the existing `query()`/`ask()` Message-union into a flat, JSON-safe, string-tagged event union (text_delta, reasoning_delta, message, tool_dispatch, tool_result, usage, notice, approval_request, ask_request, turn_started, turn_done). The TUI keeps its React fan-out for now but can be re-expressed as a thin reducer over this event; serve switches from the dead-end `queryRuntimeWithStreaming(tools:[])` to running the REAL engine and serializing via `toWireEvent`. This deletes the divergence (one vocabulary), makes serve actually useful, and is the ONLY hard prerequisite for any desktop GUI.

B. THE CONTROLLER ABSTRACTION + DESKTOP GUI (expensive, defer): a `SessionController` class wrapping QueryEngine that exposes verbs (submit/cancel/approve/answer) and an event subscription, decoupled from React. This is the Reasonix Controller analog. It is genuinely large because DeepCode's permission flow (useCanUseTool.tsx) is React-state-bound, and the TUI's 8-callback fan-out + tool-confirm queue + streaming-tool-use accumulation is deeply entangled with REPL.tsx (4883 lines). A desktop GUI on top (Tauri preferred over Electron for bundle size; consumes serve SSE in-process or over loopback + a React frontend reusing the event types) only becomes worthwhile if serve is a real product surface AND there's demand for a non-terminal UI. For a DeepSeek-native terminal coding assistant whose entire value prop is the Ink TUI, that demand is unproven.

Recommendation: build A now (small), explicitly DEFER B until a product trigger exists (a paying integration that needs HTTP/SSE, or concrete desktop demand). A is reusable even if B never happens — it makes serve real, enables an SDK/automation story, and is the de-risking step Reasonix did first.

**Acceptance:**
- A single DeepCodeEvent union exists (src/cli/events/wireEvent.ts) and is the ONLY event vocabulary serve emits; the third ad-hoc shape in turnRunner.mjs's mapRuntimeEvent is deleted.
- toWireEvent() is a pure, unit-tested function with coverage of every Message-union variant the TUI handles; tombstone/summary correctly drop to null.
- Serve mode runs the REAL agent loop (tools + permissions + hooks via ask()), not queryRuntimeWithStreaming(tools:[]); a serve turn produces tool_dispatch/tool_result/approval_request events over SSE.
- Serve is reachable via `deepcode --serve --http` behind a feature flag, with bearer-token auth intact, and ships dark by default.
- TUI behavior is UNCHANGED for end users (Phase 1 does not regress REPL.tsx); if Phase 1d is done, a parity test proves the reducer path matches the legacy fan-out.
- A written go/no-go on Phase 2 (controller + desktop) with the explicit product trigger that would flip it — captured in the workstream notes, not started speculatively.

**Risks:**
- Over-investment risk (primary): the full controller+desktop bet is XL effort for a terminal-first product with no proven non-TUI demand. Reasonix's tri-frontend design is a from-scratch architectural choice; retrofitting it onto a 4883-line REPL.tsx + React-bound permissions is far costlier than greenfield. Mitigation: do only Phase 0/1, defer the rest behind a product trigger.
- Permission-flow entanglement: useCanUseTool.tsx (src/hooks/useCanUseTool.tsx:33) resolves tool approvals through React setState (setToolUseConfirmQueue). Any controller must lift this into a transport-neutral registry — this is the single hardest part of Phase 2 and the main reason it's L+. Phase 1b sidesteps it with a serve-local non-interactive bridge, but a unified controller cannot.
- Serve turning 'real' increases attack surface: it will execute tools (Bash/Edit) over HTTP. Bearer auth (serve/auth.mjs) + localhost-only default binding (http.mjs DEFAULT_HTTP_HOST 127.0.0.1) must be preserved, and the default no-answer approval policy must be DENY. Treat exposing serve as a security-sensitive change (per global instructions: review SECURITY posture before enabling remote execution).
- Maintaining two mappers during transition: until Phase 1d, handleMessageFromStream and toWireEvent both encode event classification. Drift risk — mitigated by the parity test, but adds a maintenance touchpoint.
- Tauri vs Electron lock-in (Phase 2 only): Tauri's webview is OS-dependent (WebKit/WebView2), causing platform rendering differences — same class of issue Reasonix calls out for Wails. Not a near-term risk since deferred.

- No hard dependency on the i18n workstreams currently in flight (P2.10) — this touches different files (cli/serve, cli/events) and can proceed in parallel.
